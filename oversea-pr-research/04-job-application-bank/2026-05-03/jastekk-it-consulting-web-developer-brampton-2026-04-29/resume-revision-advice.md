## Goal
- Improve resume alignment for this posting without inventing web-design or client-facing experience that is not already supported.
- Make the resume look closer to a practical web application builder who can handle coding, testing, documentation, and requirements-related work.

## Working Resume Base
- Reference draft: `01-core/01-documents/resume-reference-owp.md`
- Use this advice as targeted edits to that draft, not as a full rewrite.

## Main Positioning Angle
- Position the profile as a web application developer with strong overlap in React, APIs, backend integration, database-backed systems, requirements analysis, testing, troubleshooting, and technical documentation.
- Do not position the profile as a visual designer or as someone with strong client-presenting experience unless there is real evidence.

## Section-By-Section Revision Plan

### 1. Summary Section
- Section to revise: `## Professional Summary`
- Why: the posting is broad and web-delivery oriented, so the summary should sound more web-facing and implementation-focused.

**Current direction**
- Strong on enterprise systems and backend reliability
- Less clear on web application delivery, testing, and requirements-to-build flow

**Revised version**
- `Software Developer with 5+ years of experience building and supporting web and enterprise applications across frontend development, backend services, API integration, and database-driven systems. Experienced in translating requirements into maintainable application features, supporting testing and troubleshooting, and improving reliability through practical documentation and structured problem-solving.`

### 2. Technical Skills Section
- Section to revise: `## Technical Skills`
- Why: the posting emphasizes websites, layout and flow, code integration, testing, and content-oriented web delivery, so the skills section should surface the most relevant overlap earlier.

**Suggested direction**
- Move `React`, `TypeScript`, `JavaScript`, `RESTful API design and integration`, and `requirements gathering and analysis` into more visible positions.
- Keep database, backend, and troubleshooting signals because they support the testing and quality-control side of the role.

**Suggested revised emphasis**
- `Core Competencies: requirements gathering and analysis, web application development, API integration, technical documentation, troubleshooting, testing support`
- `Frontend And Web: React, TypeScript, JavaScript, Angular, HTML/CSS if truthful from past work, Material UI`
- `Backend And Integration: RESTful API design and integration, Spring Boot, NestJS, Express.js, database-driven systems`
- `Quality And Delivery: debugging, system reliability, deployment support, structured logging`

### 3. Current Role Bullets
- Section to revise: `System Developer`
- Why: this is the strongest current evidence for web-based delivery, requirements work, integration, and maintainability.

**Best bullets to strengthen**
- Emphasize that the system is a `web-based` platform
- Keep the React frontend work highly visible
- Keep requirements analysis and documentation visible
- Keep stability and debugging work because the posting includes testing, quality control, and incident handling

**Suggested direction**
- Strengthen wording around:
  - web-based system enhancement
  - frontend feature development
  - API integration
  - requirements analysis
  - documentation
  - debugging and partial-failure handling

### 4. Previous Role Bullets
- Section to revise: `Analyst Programmer`
- Why: this role adds breadth across frontend, backend, testing, databases, and support.

**Best evidence to pull forward**
- frontend and mobile application work
- backend microservices and integration
- stress testing / system stability work
- technical support and problem solving
- database structure work

**Suggested direction**
- Reword the most relevant bullets so they sound closer to:
  - application feature delivery
  - integration work
  - testing and stability checks
  - support for business workflows
- Keep this practical and avoid making it sound like pure design work.

### 5. Education / Supporting Signals
- Why: this posting only requires secondary school, so your degree is already more than enough.
- Keep the degree concise.
- Do not use too much space on education if it pushes more relevant web/application experience down the page.

## What To Highlight Most

## What Not To Overclaim

- multidisciplinary team leadership
- incident management ownership if your experience is more support-oriented than formal incident lead work

## Main Resume Risks For This Posting
- The current resume reads more like enterprise software / backend support than broad web-production
- The posting includes content/layout/mockup language that your current evidence may not strongly support
- The role appears communication-heavy, so the resume should sound clear, practical, and stable rather than overly technical or abstract
